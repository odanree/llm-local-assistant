## 🎯 Feature Overview

Move from **Passive Execution** (waiting for commands to finish) to **Active Monitoring** (interacting with commands as they run).

### Current Limitations (v2.11.0)
- ❌ Commands execute completely, then return full stdout/stderr
- ❌ No real-time log feedback to user
- ❌ No interactive stdin support (prompts hang, causing "Retry Trap")
- ❌ No process cancellation (kill switch)
- ❌ Executor loses state on npm/interactive command failure
- ❌ No ability to pause plan for user permission
- ❌ File changes during suspension not detected

### Target Capabilities (v2.12.0)
- ✅ Real-time streaming of stdout/stderr to VS Code Output Channel
- ✅ Interactive stdin support for terminal prompts (npm, yarn, git)
- ✅ Process management with cancellation tokens
- ✅ Graceful timeout handling (>30s auto-cancel)
- ✅ Executor suspend/resume on user permission requests
- ✅ PlanState.SUSPENDED_FOR_PERMISSION state machine
- ✅ State integrity verification before resuming
- ✅ File conflict detection while executor is paused
- ✅ Zero "Retry Trap" loops (user-controlled resume instead)

```typescript
enum PlanState {
  IDLE = 'idle',
  PLANNING = 'planning',
  EXECUTING = 'executing',
  SUSPENDED_FOR_PERMISSION = 'suspended_for_permission', // NEW
  PAUSED = 'paused',
  COMPLETED = 'completed',
  FAILED = 'failed'
}

interface SuspendedExecutionState {
  stepIndex: number;
  step: ExecutionStep;
  planId: string;
  timestamp: number;
  remainingSteps: ExecutionStep[];
  context: {
    lastStdout: string;
    lastStderr: string;
    fileSnapshot: Record<string, string>; // file path → hash
  };
}
```

**Executor Loop Logic**:
```typescript
// In executePlan() main loop
for (let i = 0; i < plan.steps.length; i++) {
  const step = plan.steps[i];

  try {
    const result = await executeStep(step);

    if (result.requiresUserInput) {
      // Suspend execution
      this.state = PlanState.SUSPENDED_FOR_PERMISSION;
      this.suspendedState = {
        stepIndex: i,
        step,
        planId: plan.id,
        timestamp: Date.now(),
        remainingSteps: plan.steps.slice(i + 1),
        context: {
          lastStdout: result.stdout,
          lastStderr: result.stderr,
          fileSnapshot: await this.captureFileHashes()
        }
      };

      // Persist to codebaseIndex for crash recovery
      codebaseIndex.setPlanSuspension(plan.id, this.suspendedState);

      // Emit event to UI
      emitEvent('executor:suspended', {
        stepIndex: i,
        message: `Step requires input: ${step.label}`,
        resumable: true
      });

      // Block here until user calls resume()
      return PlanState.SUSPENDED_FOR_PERMISSION;
    }
  } catch (error) {
    // ... existing error handling
  }
}
```

**Resume Logic**:
```typescript
async resume(userInput: string): Promise<void> {
  if (this.state !== PlanState.SUSPENDED_FOR_PERMISSION) {
    throw new Error('No suspended execution to resume');
  }

  // Verify state integrity before resuming
  const integrity = await this.verifyStateIntegrity();
  if (!integrity.valid) {
    throw new Error(`State corrupted: ${integrity.reason}`);
  }

  // Send user input to process
  this.currentProcess?.sendInput(userInput);

  // Continue execution
  this.state = PlanState.EXECUTING;
  // Re-enter main loop for remaining steps
}
```

### Milestone 4: State Integrity & Safety Checkpoints (Day 2-3)
**Goal**: Verify file consistency before resuming from suspension

**Deliverables**:
- `verifyStateIntegrity()` method with atomic write verification
- File hash comparison for all WRITE steps preceding RUN
- Conflict detection (user edited file while executor paused)
- User prompts for conflict resolution
- Integrity check tests

**Coverage Gain**: +0.2%

**Key Implementation**:
```typescript
interface IntegrityCheckResult {
  valid: boolean;
  reason?: string;
  conflicts?: {
    filePath: string;
    expectedHash: string;
    actualHash: string;
    action: 'block' | 'overwrite' | 'skip';
  }[];
}

async verifyStateIntegrity(): Promise<IntegrityCheckResult> {
  if (!this.suspendedState) {
    return { valid: true };
  }

  const previousWrites = this.suspendedState.step
    .findPrecedingSteps('WRITE');

  const conflicts = [];

  for (const writeStep of previousWrites) {
    const expectedHash = this.suspendedState.context.fileSnapshot[writeStep.filePath];
    const actualContent = await fs.readFile(writeStep.filePath, 'utf8');
    const actualHash = hashFile(actualContent);

    if (expectedHash !== actualHash) {
      conflicts.push({
        filePath: writeStep.filePath,
        expectedHash,
        actualHash,
        action: 'block' // User must choose
      });
    }
  }

  if (conflicts.length > 0) {
    return {
      valid: false,
      reason: `${conflicts.length} files changed while executor was suspended`,
      conflicts
    };
  }

  return { valid: true };
}
```

**Conflict Resolution Options**:
1. **Block Resume**: "Files changed. Please resolve conflicts and try again."
2. **Re-run WRITE Steps**: "Re-apply all preceding WRITE steps and resume RUN."
3. **Skip Integrity Check**: "Resume anyway (advanced, risky)."

**Test Scenarios**:

## 🎯 Key Architectural Decisions

### 1. Suspend Persistence Strategy
- **Choice**: VS Code workspace state + codebaseIndex
- **Why**: Survives extension restart, scoped to workspace, debuggable
- **Risk Mitigation**: File hash verification before resume

### 2. Resume Trigger
- **Choice**: Automatic on sendInput(), explicit `/resume` for manual approval
- **Why**: Seamless for interactive prompts, explicit control for approval workflows
- **Safety**: State integrity check always runs before continuing

### 3. Conflict Resolution Policy
- **Choice**: Block resume if files changed, offer re-run from step
- **Why**: Prevents data loss, gives user agency
- **Alternative**: Add "Force Resume" option for advanced users

### 4. Prompt Detection
- **Strategy**: Regex patterns + heuristics (ends with `?`, `>>`, `[Y/n]`, etc.)
- **Fallback**: User can manually type into OutputChannel if auto-detection fails
- **Test**: All major package managers (npm, yarn, pnpm, git)

## 🔄 The "Yield & Resume" Logic Flow

This is the core innovation that solves the "Retry Trap" problem:

```
BEFORE v2.12.0:
┌─────────────────────────────────────────┐
│ RUN: npm install (requires y/n input)   │
├─────────────────────────────────────────┤
│ Process hangs waiting for stdin         │
│ Executor hangs (no sendInput support)   │
│ User sees: "Retrying (1/2)..."          │
│ Eventually fails, plan aborts ❌        │
└─────────────────────────────────────────┘

AFTER v2.12.0:
┌─────────────────────────────────────────┐
│ RUN: npm install (requires y/n input)   │
├─────────────────────────────────────────┤
│ Process outputs prompt in real-time      │
│ Executor detects prompt, suspends       │
│ Executor state saved to codebaseIndex   │
│ User sees: "Waiting for input: y/n"     │
│ User clicks "Yes" → sendInput('y')      │
│ Executor resumes from EXACT same step   │
│ Process continues ✅ → Plan succeeds    │
└─────────────────────────────────────────┘
```

### Detailed Implementation Notes

**Step 1: Detect Requirement for Input**
```typescript

// Catch patterns that indicate user input needed:
const NEEDS_INPUT_PATTERNS = [
  /npm ERR!.*\(y\/n\)/i,           // npm yes/no

  /Select a version/,              // version selectors
  />>/,                            // git rebase
  /Password:/,                     // ssh/sudo
  /Proceed\?/,                     // generic proceed
];

// Set flag on result

  stdout: '...',
  stderr: '...',

```


```typescript

  // Verify no files were modified while suspended
  const integrity = await this.verifyStateIntegrity();
  if (!integrity.valid) {
    emitEvent('executor:integrity_failed', {
      conflicts: integrity.conflicts
    });
    // User must resolve conflicts before continuing
    return;
  }

  // Send input to process
  this.currentProcess?.sendInput(userInput);

  // Continue execution
  this.state = PlanState.EXECUTING;

  // Loop continues in next iteration with remaining steps
}
```

### Problem Solved: The "Retry Trap"

**Before**: npm install fails → auto-retry → fails again → fails plan
**After**: npm install needs input → pause → user responds → continue → success

This is the key UX improvement of v2.12.0.
